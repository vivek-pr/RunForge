# RunForge

RunForge is a Kubernetes operator project (Kubebuilder-based) for managing `AIJob` custom resources.

This repository includes:
- production-style scaffolding and CI/tooling
- local-first Kind automation
- `AIJob` API (`v1alpha1`) with a realistic spec/status shape
- controller MVP that creates/owns a Kubernetes Job per `AIJob`
- mandatory developer tracking artifacts under `developer/`

## Current Status

- Kubebuilder project initialized (`domain: runforge.io`, `repo: github.com/vivekpradhan/runforge`).
- API created: `group=runforge`, `version=v1alpha1`, `kind=AIJob`.
- CRD install/uninstall flow works via `make install` / `make uninstall`.
- Controller MVP implemented:
  - creates Job if missing
  - sets owner reference
  - emits `JobCreated` event
  - updates `status.jobName`, `status.phase`, `status.observedGeneration`
  - updates lifecycle status conditions from Job state (`Pending`, `Running`, `Succeeded`, `Failed`)
  - sets `status.startTime`, `status.completionTime`, and `status.lastError` for failure paths
  - idempotent on re-apply (does not create duplicate Jobs)
- Local Kind lifecycle automated with one-command targets.

## Repository Layout

- `api/v1alpha1/`: `AIJob` API types and generated deepcopy code.
- `internal/controller/`: reconciler implementation.
- `internal/jobfactory/`: pure function that builds Kubernetes Job objects.
- `internal/status/`: lifecycle condition/status derivation from Kubernetes Job state.
- `config/`: CRD/RBAC/manager manifests generated by Kubebuilder.
- `examples/`: runnable `AIJob` examples (`aijob-success.yaml`, `aijob-fail.yaml`).
- `hack/kind/`, `hack/scripts/`: Kind cluster config and up/down scripts.
- `docs/`: API and local-dev documentation.
- `developer/`: mandatory engineering tracking logs.

## Key Make Targets

Run `make help` for the complete list. Most-used targets:

- `make kind-up`: create/use local Kind cluster (`runforge-local`)
- `make kind-down`: delete local Kind cluster
- `make kube-context`: print active kubectl context
- `make generate`: generate deepcopy code
- `make manifests`: generate CRDs/RBAC/webhook manifests
- `make install`: install CRD into current cluster
- `make uninstall`: remove CRD from current cluster
- `make run`: run controller locally against current kube-context
- `make test`: unit/envtest suite
- `make lint`: run golangci-lint

## Local Workflow

### 1) Bring up local cluster

```bash
make kind-up
kubectl get nodes
```

### 2) Install CRD and run controller

```bash
make install
make run
```

### 3) Apply sample AIJob

In another terminal:

```bash
kubectl apply -f examples/aijob-success.yaml
kubectl get aijob,jobs,pods
kubectl describe aijob aijob-success
```

### 4) Verify cleanup behavior

```bash
kubectl delete -f examples/aijob-success.yaml
kubectl get jobs,pods
```

### 5) Tear down

```bash
make kind-down
```

## AIJob API

- Full API reference: [docs/api.md](/Users/vivekpradhan/vscode/InterViewPreparation/RunForge/docs/api.md)
- Version: `runforge.runforge.io/v1alpha1`
- Resource: `aijobs`

Core `spec` fields include:
- container execution (`image`, `command`, `args`)
- env and env sources (`env`, `envFrom`)
- resources (`requests/limits` cpu/memory and optional gpu)
- retry/deadline/ttl (`restartPolicy`, `backoffLimit`, `activeDeadlineSeconds`, `ttlSecondsAfterFinished`)
- scheduling (`nodeSelector`, `tolerations`, `affinity`)
- identity (`serviceAccountName`)

Core `status` fields include:
- `observedGeneration`, `phase`, `jobName`
- `conditions`
- `startTime`, `completionTime`, `lastError`

Status lifecycle behavior:
- Pending when Job exists but is not active/terminal
- Running when Job has active pods
- Succeeded when Job has `Complete=True`
- Failed when Job has `Failed=True` (also sets `lastError`)
